# **들어가며**

유튜브로 2024년 롤드컵을 보고 있엇습니다.  경기 중 다이나믹하고 슈퍼 플레이들이 나올 때는 채팅창이 매우 불타는 모습을 보여줬습니다.  저의 눈엔  롤도 롤이지만  해당 라이브를 보고 있는 동시 접속자 수가 눈에 들어왔고  대략30만을 넘고 있었습니다.  문득 이러한 **“라이브 스트리밍 서버를 운영한다면 얼마나 많은 시스템 자원이 필요할까??? 스펙아웃 한다면 몇 대의 머신이 필요할까 라는 의문이 들었습니다.”**  그래서 고민으로 끝내는 것보단 한 번쯤은 비슷한 상황을 가정하고 트래픽을 핸들링 해보고싶었습니다.

실제 라이브스트리밍은 HLS (HTTP Live Streaming) DASH(Dynamic Adaptive Streaming over HTTP) 기술들을 활용한다고 하며, 다음과 같은 Flow를 따릅니다

```
카메라 ---> 인코더 ---> 미디어 서버 ---> 클라이언트
```

여기서 카메라 → 인코더 → 미디어 서버 까지의 구현은 목표가 아니기 때문에 제외하고 **“미디어 서버 → 클라이언트”** 까지의 Flow에 부하가 생겼을 때 라는 상황을 처리해보고자 합니다. 이 문제를 풀고자 생각했던 또 다른 이유는 유명한 [The c10k problem](http://www.kegel.com/c10k.html) 을 알고 있었기에  현대 시점에 맞게 문제를 해결해보고자 했습니다.

이 글은 **목차나 앞으로의 계획이 명확한 글은 아닙니다!** 저의 호기심을 해결하는 것이 가장 큰 목적이기에 명확한 목차가 있는 것도 아니고 결말이 정해진 글도 아닙니다. **그 때 그 때 궁금하면 더 추가하는 방향으로 나아가고자 합니다.**

# **우선, 개념정리부터**

이 문제에 대해 고민하다 보니 기본 개념들에 대한 정리가 부족하다는 생각이 들었습니다. 특히, `소켓`, `커넥션` 이란 것이 저에게 조금 모호한 느낌이었습니다.

### 소켓:

1. 소켓은 Transport Layer와 Application Layer 사이의 Interface입니다. 전송 계층(TCP, UDP)과 애플리케이션 계층(예: HTTP, FTP) 사이에서 데이터의 교환을 중개하고, 네트워크 통신을 위한 API를 제공합니다.
2. 커널 입장에서 소켓은 하나의 파일입니다. 소켓으로 들어오는 데이터는 `read()` 함수를 사용해 읽고 보내는 데이터는 `write()` 함수를 이용해 전송합니다.
3. 이 뜻은 패킷을 처리할 때 File I/O가 일어난다는 것이되며, 이를 위해 `File Discripter` 라는 리소스 또한 소켓과 커넥션을 관리하는 입장에서 중요한 포인트가 됩니다.
4. 소켓은 `ip address + port`  으로 생성되며, 하나의 머신에서 소켓은 유니크한 값입니다. `socket()` 이라는 System Call을 통해 생성됩니다.
    
    ```cpp
    include <sys/socket.h>
    
    // 성공하면 파일 디스크립터을 리턴, 에러가  발생하면 '-1'을 리턴
    // domain: 프로토콜 family를 지정해주는 변수
    // type: 어떤 타입의 프로토콜을 사용할 것인지 설정 SOCK_STREAM(TCP), SOCK_DGRAM(UDP), SOCK_RAW(사용자 정의)
    // protocal: 어떤 프로토콜의 값을 결정하는 것 0, IPPROTO_TCP(TCP), IPPROTO_UDP(UDP)
    int socket(int domain, int type, int protocol)
    ```
    
    1. `ip address + port` 이라는 조합은 네트워크의 표준 스펙에 나오는 내용이지만 실제 우리가 사용하는 WAS 구현에선 조금 다르게 동작합니다. `ip address + port` 조합이 단 하나의 유니크 값이라면 하나의 WAS는 소켓을 하나만 생성할 수 있다는 것이기 때문에 여러 클라이언트를 받을 수 없다는 의미가 되기도 합니다.
    2. 따라서 WAS는 `Listening Socket`과 `Connection Socket` 을 분리하지만 모두 동일한 `IP Address + Port` 조합으로 생성됩니다. 따라서 하나의 서버에 같은 `IP Address + Port` 조합이 여러 개 생기게 되기에, WAS는 여러 Client와 통신할 수 있게됩니다.
        1. Listening과 Connect를 분리하는 것은 클라이언트와 송/수신을 시작해도, 계속해서 다른 Client로부터 요청을 받기 위함입니다.  즉, Listening Socket은 Client로부터 연결 수락을 무한히 대기하고 **“연결 요청”**을 수락합니다. 반면, Connection Socket은  Connection의 데이터 송/수신을 처리합니다.
    3. 이러한 메커니즘에는 큰 문제가 있는데, 여러 요청을 받을 수 있기 때문에 여러 클라이언트로부터 오는 패킷들이 정확히 어떤 socket으로 가야할지 판단하지 못하게됩니다. **즉, 신뢰성이 매우 떨어집니다.** 이 문제를 해결하기 위해 커넥션을 활용하게 됩니다.

### **커넥션:**

- `<source ip:port>` + `<destination ip:port>` 의 조합으로 Network 에서 정말 유일한 값이 되며 식별자로 활용할 수 있는 값이됩니다.
- TCP는 3way hand shake를 통해 Connection을 생성하며 Connection을 이용해 신뢰성 있는 통신을 지원합니다.
- 따라서 WAS는 여러 클라이언트의 요청을 수신할 수 있는 여러 개의 Socket이 생겨도 **커넥션은 유일한 식별 값 역할을하기 때문에 안전하게 통신할 수 있습니다.**

**정리하면:**

1. **Socket은 전송 계층과 애플리케이션 계층 사이에서 여러 API를 제공해주는 인터페이스입니다.**
2. socket은 동일한 `ip:port` 조합으로 여러개 생길 수 있지만, **`Connetion`**은  **Unique하기 때문에 여러 클라이언트와 통신에서도 신뢰성을 보장할 수 있습니다.**

## 그렇다면, 한 대의 서버로는 몇개의 Connection을 가질 수 있을까?

라이브 스트리밍 서버는 Web Socket을 사용하여 Connection을 유지하는 것이 일반적입니다. 즉, 경기 시작부터 끝까지 Connection이 물리는 경우가 있다는 것이며, 입소문타서  경기가 유명해지면 경기 도중 Connection이 얼마든지 늘어날 수 있습니다. 그렇다면 한 대의 서버를 운영한다면 **최대 몇개의 TCP Connection을 맺을 수 있을까요? 한 대의 서버로 동시 접속자수 30만명이 가능 할까요??**


> **이론적으로 한 대의 서버로도 자원만 충분하다면 30만개 이상의** `Connection Socket` **을 만들어 Connection을 유지할 수 있습니다.**
{: .prompt-tip }

1. Connection은 `source ip + port` 를 포함하기 때문에 **클라이언트의 트래픽이 out bound port만 다르다면, 동일한 클라이언트 기준으로 최대 65535개의 Connection을 생성할 수 있습니다.** 
    1. 이러한 동작은 트래픽이 나갈 때 커널에서 사용하지 않는 랜덤한 port를 부여받는데 이 범위가 0~65535이기 때문입니다.
2. 즉, 클라이언트마다 65535개의 커넥션을 맺을 수 있으며, 클라이언트의 IP만 다르다면 (다른 Client) 이론적으로 **Connection은 30만개 이상 만들 수 있습니다.** 그렇다면 세상에 가장 좋은 CPU를 장착시키고 RAM 또한 수백 GB를 넣어준다면 **한 대의 서버로 무한히 많은 커넥션을 생성할 수 있을까요?** 
    
    **답은 No 입니다.**
    

**리눅스 서버에서 Socket은 File이며, File은 오픈 가능한 Limit이 정해져 있습니다.** Connection은 `Connection Socket` 에 물려있기 때문에  **Limit에 도달하면 시스템 자원은 남아 있어도 더 이상 Open할 수 있는 파일이 없게 되어 더 많은 연결을 맺지 못하게 됩니다.** 

**하나의** **Process**가 Open 가능한 파일 수는  `ulimit`이란 리눅스 명령어를 통해  확인 가능합니다. 

```bash
$ ulimit -a  // soft ulimit
$ ulimit -aH // hard ulimit

---
real-time non-blocking time  (microseconds, -R) unlimited
core file size              (blocks, -c) unlimited
data seg size               (kbytes, -d) unlimited
scheduling priority                 (-e) 0
file size                   (blocks, -f) unlimited
pending signals                     (-i) 1028770
max locked memory           (kbytes, -l) 32930388
max memory size             (kbytes, -m) unlimited
open files                          (-n) 1048576 # 최대 open 가능한 파일 수
pipe size                (512 bytes, -p) 8
POSIX message queues         (bytes, -q) 819200
real-time priority                  (-r) 0
stack size                  (kbytes, -s) unlimited
cpu time                   (seconds, -t) unlimited
max user processes                  (-u) 1028770
virtual memory              (kbytes, -v) unlimited
file locks                          (-x) unlimited
```

`ulimit`은 하나의 Process에서 Open 가능한 값을 보여준다면 시스템 전체의 설정 값은 `/proc/sys/fs/file-max` 에 설정되어 있습니다.

```bash
cat /proc/sys/fs/file-max

---
9223372036854775807
```

## 왜그럴까?? - feat.파일 디스크립터:

> 리눅스에서 File을 무한정 열지 못하는 이유는 Process들이 File에 접근하기 위해선 반드시 File Discripter로부터 부여받은 Index 번호가 필요하고, 파일 디스크립터의 Index 번호는 무한하지 않기 때문입니다. (File Table의 Size만큼 존재함)
> 

**파일 디스크립터:**

커널로부터 할당받은 0이 아닌 정수 값으로 **프로세스에서 파일 목록을 관리하는 일종의 포인터 입니다. (정확히는 파일 테이블을 참조하는 포인터) 커널은 파일에 접근하려고 하는 모든 프로세스에게 임의의 정수값 (인덱스)을 제공하여 File Access를 관리합니다.** 

이 때 사용되는 임의의 정수값 (인덱스)는 무한한 것이 아니라 **`OPEN_MAX`** 까지의 한계가 있습니다. 이는 Table이기 때문에 무한히 커지는 것을 방지하기 위함이며 시스템의 안정성을 높이기 위한 설정값입니다. 따라서 동시에 수많은 커넥션을 맺어야하는 하는 **라이브 스트리밍** **서버 입장에서는 시스템 리소스 (CPU, Memory 등등)가 넉넉해도 Connection을 Open 하는데 한계가 있다는 의미가 됩니다.**

**정리:**

- 프로세스는 파일에 접근할 때 File Discripter로 부터 고유한 번호를 부여받아야 합니다.
- 이는 한정된 수량으로 관리 됩니다.  따라서 **동시 접속자가 많고 커넥션을 유지하는 스트리밍 서버에서는  File Discripter가 부족해지는 현상으로 인해 장애가  발생할 수 있습니다.**

## 개발전, 필요한 리소스 예측 해보기

**라이브 스트리밍 서비스의 특성 정의:**

1. 라이브 스트리밍 서버는 커넥션이 일정기간 이상 유지되는 경우가 많다.
    1. 커넥션 메모리 size x 최소 접속자 수 만큼은 메모리가 유지될 것이다.
2. 갑자기 유입량이 늘어나면 순간적으로 Connection 요청이 몰릴 수 있다. (3way handshake가 급증할 수 있음)
    1. Hand Shake 요청이 급증하여 CPU 과부하가 걸릴 확률이 높습니다.
3. 지속적으로 선형적인 형태로 부하가 늘어날 수 있다.

### 메모리 예측 해보기

 Socket의 Memory Size는 운영체제의 구현과 설정에 따라 약간씩 달라집니다.  Socket은 `상태`, `옵션`, `송/수신 버퍼`, `연결 정보`( IP, Port) 정보등이 담깁니다. 특히 소켓의 많은 부분은 버퍼 사이즈가 결정하기에 버퍼 사이즈를 체크하는 것이 핵심입니다.

 버퍼 사이즈를 체크하기 위해선 Ubuntu 기준으로 `sysctl`을 사용하여 TCP 관련 커널 파라미터를 확인하여 **송/수신 버퍼 사이즈를 체크할 수 있습니다.**

```bash
sysctl net.ipv4.tcp_rmem # Recive
sysctl net.ipv4.tcp_wmem # Write
sysctl net.ipv4.tcp_mem 
---
# 최소값, 기본값, 최대값을 바이트 단위로 출력
net.ipv4.tcp_rmem = 4096	131072	6291456 # Recive
net.ipv4.tcp_wmem = 4096	16384	4194304 # Write
net.ipv4.tcp_mem = 3079083	4105445	6158166
```

**`tcp_rmem` 및 `tcp_wmem:`**

- Byte 단위로 출력되며, 각각의 TCP 소켓이 갖는 개별 수신 및 송신 버퍼의 크기를 설정합니다.

**`tcp_mem:`**

- 시스템 전체의 TCP 메모리 사용을 관리합니다. 이는 시스템의 TCP 버퍼 전체에 대한 메모리 사용량을 제한하고, 시스템 자원과 TCP 네트워크 트래픽 간의 균형을 맞추는 데 사용됩니다.

TCP 소켓의 메모리는 `rmem`, `wmem` 버퍼 사이즈에 많은 영향을 받습니다. 따라서 기본 사이즈를 기준으로 생성한다 가정 했을 때 얼만큼의 메모리가 필요할지 계산 해보겠습니다. 

> 실제 커널은 Auto-Tuning이란 메커니즘이 존재하고 이 메커니즘에 의해  송/수신 버퍼 사이즈는 동적으로 조절됩니다.
> 

### **메모리 요구사항 계산**

1. **수신 버퍼 (tcp_rmem)**:
    - 기본값을 사용한다고 가정하면: 131072 바이트 (약 128 KB)
2. **송신 버퍼 (tcp_wmem)**:
    - 기본값을 사용한다고 가정하면: 16384 바이트 (약 16 KB)

### **커넥션 당 메모리 사용량**

- 수신 버퍼: 128 KB
- 송신 버퍼: 16 KB
- 기타 TCP 연결 관련 구조체: 약 1 KB (추정치)

> 총 커넥션 당 메모리: 128 KB + 16 KB + 1 KB = 145 KB
> 

### **30만 동시 접속에 대한 총 메모리 요구사항**

> 30만 * 145 KB = 43,500,000 KB ≈ 42.5 GB
> 

**대략적으로 42.5 GB가 필요할 것으로 예측됩니다.** 여기서 수신 버퍼 사이즈가 128Kb로 기본 값으로 설정되어 있는데 스트리밍 서버의 특성을 봤을 때 실제로 필요로 하는 더 적을 것 같습니다. 그렇기에 tcp_rmem 사이즈를 줄이는 튜닝을 통해 8KB로 기본 값을 줄인다면

```bash
net.ipv4.tcp_rmem = 4096 8192 16384
```

1. **수신 버퍼**: 8KB (새로운 기본값)
2. **송신 버퍼**: 16KB (이전과 동일)
3. **기타 TCP 연결 관련 구조체**: 1KB (추정치)

커넥션 당 새로운 메모리 사용량은  (8KB + 16KB + 1KB = 25KB) 이고 이것이 30만개 필요하니 30만 * 25KB = 7,500,000KB  해서 대략적으로 **7.15GB** 만큼 필요하지 않을까합니다. 이는 추후 Load Test 시간에 검증해보도록 하겠습니다.

### 결론

- `tcp_rmem` 커널 파리미터를 수정하여 대략적으로 **7.15GB 필요하지 않을까? 합니다. (추후 Load Test 를 통해 파악하기)**
- tcp 소켓은 주로 송/수신 버퍼가 메모리의 대부분을 사용합니다.
- 수신 버퍼의 size가 비교적 큽니다. 하지만 라이브스트리밍 서버의 특성을 고려했을 때 이 부분을 줄이는 것으로 리소스를 약 35GB를 줄일 수 있습니다.

## CPU 예측 해보기

30만명의 동시접속자를 가용할 수 있어야 한다는 것은 최종적으로 30만번의 **`HandShake`**가 발생하는 것과 동일합니다. 이 때 트래픽은 크게 2가지 패턴이 있을 것 같습니다.

1. **서서히 유저가 늘어나다 하이라이트 때 peek를 찍는 경우 (선형적인 증가)**
2. **‘방송 시작’** or **‘게임 시작’** or **‘한타’**  와 특정 시간 때 급격히 늘어나는 경우 (순간적인 트래픽이 몰리는 경우)

특히 2번의 경우, TCP HandShake 요청이 급증하여, **CPU가 급격히 튈 것으로 추측됩니다. 그렇다면 왜 CPU 사용량이 높아질까요?**

### CPU 사용량이 높아지는 이유

1. **시스템 콜:** 핸드셰이크 과정에서 여러 번의 시스템 콜이 발생합니다. 이는 사용자 모드와 커널 모드 간의 전환을 유발하여 CPU 사용량을 증가시킵니다
2. **상태 관리:** TCP 연결의 상태를 관리하고 업데이트하는 과정에서 CPU 리소스가 소모됩니다.
3. **타이머 및 타임아웃 처리:** 핸드셰이크 과정에서 여러 타이머와 타임아웃을 관리해야 하며, 이는 추가적인 CPU 사용을 유발합니다.
4. **패킷 생성 및 처리:** `SYN`, `SYN-ACK`, `ACK` 패킷을 생성하고 처리하는 과정에서 CPU 리소스가 사용됩니다.
5. **네트워크 스택 처리:** TCP/IP 스택을 통한 패킷 처리 과정에서 CPU 사용량이 증가합니다.

Hand Shake 과정을 자세히 나열하지는 않았지만 Hand Shake를 하는 동안 위의 과정들이 발생하기에 대량의 동시 연결 요청이 있을 경우, 각 연결에 대한 핸드셰이크 처리가 병렬로 이루어져 CPU 부하가 급증할 수 있습니다.

### 어떤 문제가 있을까?

1. **처리 지연:** 방송 도중 스트리밍 서버의 CPU가 과부하 상태가 되면 **스트림 처리, 인코딩/디코딩, 패킷 처리 등의 작업이 지연됩니다. ← 사용자는 버퍼링을 많이 경험하게 됩니다.**
2. **연결 처리 능력 저하**: 새로운 클라이언트 연결 요청을 처리하는 능력이 떨어져 방송 참여 과정에 지연이 발생합니다. 최악의 경우, 커넥션을 맺지 못해 보고싶은 경기를 보지 못할수도 있습니다.
3. **버퍼 관리 어려움**: CPU 부족으로 인해 스트리밍 버퍼를 효율적으로 관리하기 어려워집니다.
    1. 특히, 버퍼의 경우 **메모리 할당/해제** 지연이 발생하고, **버퍼가 많아짐에 따라 컨텍스트 스위칭 비용이 많이 증가하게될 것으로 보입니다!**

### 필요한 CPU는?

1. 이 부분은 개념을 정리했지만 변수가 많기 때문에 어렵습니다. 특히 운영 환경에 따라 변수가 너무 많다 생각합니다. 따라서 명확한 수치를 계산하기는 어렵지만 대략적으로 안정적인 서비스를 위해 vCPU 16개 정도를 기준으로 시작해보자 합니다.
    
    **이렇게 생각한 이유는?**
    
    - 우선, CPU의 처리량은 클럭과 코어에 영향을 받습니다! 여기서 클럭은 단일 스레드의 연산량을 높일 때 적절하고 병렬처리에 대한 특화는 코어 수를 높이는 것이 유리합니다.
    - 이번 작업에서는 Scale Out 을 통한 처리량 높이는 것이 목적이기에 cpu의 코어 수를 높이는 방향을 선택습니다.

## 네트워크 대역폭 문제는 없을까?

> 네트워크 대역폭 문제는 이번 Task에서 제외입니다. 이번 Task의 목적은 커넥션을 맺는 행위 자체를 목표하고 부하가 높아질 때마다 서버는 어떻게 동작하는지 모니터링하는 것이 목적이기 때문입니다. 따라서 이 파트는 단순 계산을 통해 리소스를 예측하는 것이 목적입니다.

추가로 해당 연산에서는 데이터 압축은 고려하지 않았습니다.
> 

메모리와 CPU를 넉넉히 다 준다고 해서 모든 것이 준비된 것은 아닙니다. 우리는 스트리밍 서버인만큼 네트워크 대역폭 또한 처리량에 많은 영향을 끼칩니다!  커넥션이 많아지면 그만큼 해당 서버의 NIC의 사용량이 늘어나기 때문에 병목이 생깁니다.

**스트리밍 서버의 대역폭이 줄어들게될 경우:**

1.  전송량이 떨어지기 때문에 화질 저하 또는 Latency가 늘어나 버퍼가 심해지는 등의 문제가 발생합니다. **따라서 우리는 서버의 네트워크 대역폭을 다 사용하는 경우를 고려해야 합니다.** 

### 개념

> 동영상 대역폭 예측을 위해선 `비트 레이트` 개념이 필요합니다.
> 

**비트 레이트란:**

- 화면이 뭉개지지 않는, 손실 없는 원본 화질을 위한 비트레이트의 최소값
- 초당 비트 수(비트/초 또는 bps)를 사용하여 계산

**적정 비트레이트 계산식:** 해상도(가로*세로) * 초당프레임(fps) / 바이트(8bit) = 적정 Bps

**동영상 대역폭 계산식:** 적정 비트레이트 * 동시접속자

### 예측 해보자

> 롤 라이브 스트리밍이 720p 정도의 이미지를 24fps로 제공한다고 가정했을 때, 대략 초당 전송량이 나오고 대역폭을 계산할 수 있습니다.
>

**연산 해보기:**

1. **적정 비트 레이트: 1280 * 720 * 24 / 8  =  2764800 bps**
2. 필요 대역폭 계산:  (**1280 * 720 * 24 / 8** ) * 30만 (동시 접속자 수) =  199,065,600 Mbps

이는 곧  199,065.6 Gbps를 필요로하는 것이며 엄청나게 감당하기 힘든 수준의 대역폭이 필요합니다. 따라서 보통은 엄청나게 고도화된 압축 기술을 사용합니다.
